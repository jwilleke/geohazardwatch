# Deployment Architecture

How a change in `jwilleke/ngdpbase` becomes a running container behind
`https://geohazardwatch.com`. Five hops, each owned by a different workflow
or system.

## The chain at a glance

```
┌────────────────────────────┐
│ 1. ngdpbase release        │  git tag v3.39.3 on jwilleke/ngdpbase
│    (upstream repo)         │  → docker-build.yml builds & pushes
└────────────┬───────────────┘
             ▼
┌────────────────────────────┐
│ 2. ngdpbase base image     │  ghcr.io/jwilleke/ngdpbase
│    (registry)              │  tags: 3.39.3, 3.39, 3, latest
└────────────┬───────────────┘
             ▼
┌────────────────────────────┐
│ 3. ARG bump in Dockerfile  │  ARG NGDPBASE_VERSION=3.39.3
│    (this repo, via PR)     │  Renovate (auto, when working) or manual PR
└────────────┬───────────────┘
             ▼
┌────────────────────────────┐
│ 4. geohazardwatch image    │  ghcr.io/jwilleke/geohazardwatch
│    (registry)              │  tags: 1.2.26, 1.2, 1, latest
└────────────┬───────────────┘
             ▼
┌────────────────────────────┐
│ 5. Flux / k8s on prod      │  pulls new image, restarts pod
│    → geohazardwatch.com    │  Cloudflare fronts the public hostname
└────────────────────────────┘
```

## Hop 1 — `ngdpbase` upstream release

**Repo:** `jwilleke/ngdpbase`
**Trigger:** `git tag v*` on the `master` branch
**Workflow:** `.github/workflows/docker-build.yml` in ngdpbase
**Contract:** [`jwilleke/ngdpbase/RELEASES.md`](https://github.com/jwilleke/ngdpbase/blob/master/RELEASES.md)

Every `v*` tag pushes to `ghcr.io/jwilleke/ngdpbase` with four moving tags
generated by docker `metadata-action`:

| Tag form  | Example  | What it tracks                |
|-----------|----------|-------------------------------|
| full      | `3.39.3` | exact release                 |
| minor     | `3.39`   | latest patch in that minor    |
| major     | `3`      | latest minor/patch in major 3 |
| `latest`  | `latest` | newest release on default br. |

This repo pins to the **full** tag form (`3.39.3`) so the base image is
reproducible build-to-build. The shorter forms exist for callers that want
to track a minor or major series automatically — see Renovate notes below.

## Hop 2 — Base image in GHCR

**Image:** `ghcr.io/jwilleke/ngdpbase`
**Visibility:** public package on GHCR (jwilleke org)
**Consumed by:** this repo's `Dockerfile` line 12 — `FROM ghcr.io/jwilleke/ngdpbase:${NGDPBASE_VERSION}`

No work happens at this hop. It is just the durable artifact the next
hop reads from.

## Hop 3 — Bumping `NGDPBASE_VERSION` in this repo

**File:** `Dockerfile` — `ARG NGDPBASE_VERSION=<x.y.z>` on line 11
**Intended trigger:** Renovate, every 6 hours (`.github/workflows/renovate.yml`)
**Fallback:** manual PR editing the ARG

Renovate is configured in `renovate.json` to:

- detect the `ghcr.io/jwilleke/ngdpbase` dep via the standard `dockerfile` manager
- **auto-merge** minor and patch bumps (`platformAutomerge: true`)
- open a **needs-review** PR for major bumps

When working, the operator sees nothing — PRs open, CI passes, they
auto-merge, and the next hop fires. When **not** working, the base image
drifts silently. See **Known gap** below.

### Known gap — Renovate is not currently auto-firing this hop

`#66` tracks the active bug: the dockerfile manager extracts the dep but
never opens a bump PR. The two leading hypotheses are:

1. GHCR tag lookup needs auth via a `hostRules` entry in `renovate.json`
2. Multi-tag publishing confuses semver picking (without explicit `versioning`,
   Renovate may grab the shortest tag, e.g. `3`, and decide the pin is current)

Until #66 lands, **manual PRs are the operator's responsibility** when an
ngdpbase release matters (e.g., a footer-visible version bump, a security
fix, a feature this repo needs).

History: #54 bumped 3.14.5 → 3.24.4 manually (with the now-removed
customManager). #62 dropped that customManager. #65 bumped 3.24.4 → 3.39.3
manually after the drift was noticed via the live footer.

## Hop 4 — Building and publishing the geohazardwatch image

Two workflows chain here.

### 4a. `auto-tag.yml` — cut a patch tag

**Trigger:** push to `main` touching `addons/**`, `Dockerfile`, `package.json`, or `package-lock.json`
**Self-skip:** commits whose message starts with `chore: release` (its own bump commits)
**Token:** `RELEASE_PAT` (NOT the default `GITHUB_TOKEN`) — required so the
pushed tag is treated as a user push and **triggers** `publish-image.yml`.
The default token would not.
**What it does:**

1. Runs `npm run version:bump` (`tsx src/utils/version.ts patch`)
2. That updates `package.json`, `addons/geohazardwatch/index.js`, and
   prepends a section to `CHANGELOG.md`
3. Commits as `chore: release vX.Y.Z`
4. Tags `vX.Y.Z` and pushes

### 4b. `publish-image.yml` — build and ship

**Trigger:** push of a `v*` tag
**What it does:**

1. `docker/build-push-action` builds `./Dockerfile` (the new
   `NGDPBASE_VERSION` is baked in here)
2. `metadata-action` generates the same four-tag fan-out as ngdpbase
   itself (full / minor / major / `latest`)
3. Pushes to `ghcr.io/jwilleke/geohazardwatch`
4. **Smoke test:** starts the freshly-built container with
   `HEADLESS_INSTALL=true`, waits for `/healthz`, hits an HTTP probe,
   cleans up. A red smoke test fails the workflow and **the image is still
   pushed** (push happens before smoke) — see "Failure modes" below.
5. **Trivy scan:** publishes vulnerability findings to the security tab.

Typical wall-clock: ~2 minutes for build + smoke + scan.

## Hop 5 — Flux pulls and rolls

**System:** Flux on the production Kubernetes cluster (separate repo, not
in-tree here).
**Trigger:** Flux image-reflector controller polls
`ghcr.io/jwilleke/geohazardwatch` on its configured interval.
**What happens:** when a newer tag matching the configured policy appears
(typically `latest` or a semver range), the image-automation controller
patches the cluster manifests, Flux reconciles, and the geohazardwatch
pod is recreated with the new image.

**Public surface:** `https://geohazardwatch.com` is fronted by Cloudflare
(see closed epic #14 — Cloudflare Tunnel). The Cloudflare layer is
caching for static assets but `cf-cache-status: DYNAMIC` for the wiki
pages, so the version footer reflects the running pod with no
intermediate cache to bust.

**Observability:** the user-visible signal that the chain has completed
is the footer at `https://geohazardwatch.com/view/SystemInfo`, which
renders `GeoHazardWatch v{$version}` where `{$version}` is the **ngdpbase
host** version — not this repo's version. (This is why the footer says
`v3.39.3` after a base-image bump, not `v1.2.26`.)

## End-to-end timing (observed)

For the #65 unblock (manual base-image bump):

| Step                                          | Elapsed       |
|-----------------------------------------------|---------------|
| PR open → CI green                             | ~25 s         |
| Merge → `auto-tag.yml` cuts v1.2.26            | ~20 s         |
| Tag push → `publish-image.yml` finishes        | ~2 m 8 s      |
| Image in GHCR → live footer shows new version  | a few minutes |

**Total: roughly 5 minutes** under good conditions.

## Failure modes and where to look

| Symptom                                       | Likely hop | Where to check                                                |
|-----------------------------------------------|------------|----------------------------------------------------------------|
| Live footer behind ngdpbase latest tag        | 3          | This repo's `Dockerfile` — is `NGDPBASE_VERSION` current?      |
| No Renovate PR despite drift                  | 3          | `.github/workflows/renovate.yml` run logs; see #66             |
| Tag cut but no image published                | 4b         | `publish-image.yml` run for that tag                           |
| Image pushed but smoke test red               | 4b         | The smoke-test steps in the publish-image run                  |
| Image green, pod still running old version    | 5          | Flux logs / `kubectl get pod -n <ns>` on prod cluster          |
| Pod has new image but page still shows old    | 5          | Cloudflare cache (rarely — dynamic pages are not cached)       |
| Tag pushed but `publish-image.yml` didn't run | 4a → 4b    | Check token: `RELEASE_PAT` (not default GITHUB_TOKEN)          |

## Manual operator actions

**Force an ngdpbase bump** (when Renovate isn't firing): open a one-line
PR editing `Dockerfile`. Merge triggers the rest automatically.

**Force a republish** without a code change: `workflow_dispatch` on
`publish-image.yml` with an explicit `tag` input — useful if a previous
build was bad or if a manifest change requires a fresh container.

**Force Flux reconciliation:** outside this repo. Flux's standard CLI
(`flux reconcile image update` or pod restart) on the production cluster.
